# Fitness Agent Requirements Analysis Pack

## 1) Problem Statement

你要构建一个私人健身教练Agent，能够长期跟踪训练进度并给出可执行建议。

当前核心挑战：

- 训练建议缺乏稳定的数据闭环
- 训练负荷与恢复状态未形成自动调节逻辑
- 缺少长期留存机制（提醒、周报、反馈循环）

## 2) MVP Scope

MVP目标（4-6周）：

1. 记录训练与恢复数据
2. 自动生成下次训练建议
3. 每日提醒 + 每周周报
4. 用户可查看趋势并追溯建议原因
5. 提供一个简洁前端页面用于查看和编辑训练状态
6. 建立体测记录库（体测单图片/PDF、OCR提取结果、可纠正电子表单、身体变化图表）

MVP不做：

- 动作姿态识别
- 复杂视觉化前端（重设计、重动画、复杂BI报表）
- 深度学习预测模型

## 2.1) 前端与体测记录库范围（新增）

定位：

- 消息界面负责高频交互（提醒、问答、快速记录）
- Web页面负责结构化查看和编辑（状态总览、历史回看、体测记录库管理）

前端信息架构（MVP）：

1. 今日面板

- 今日是否训练
- 下一次训练时间
- 今日建议（主项/负荷/组次/理由）
- 目标追踪卡（当前体重、目标体重、剩余差距、目标截止日期）
- 肌肉增长目标卡（当前肌肉量、目标肌肉量、剩余差距）
- 进度反馈（当前增长曲线：健康/过慢/超额）

2. 训练计划面板

- 支持可配置周期模板（7/14/21/28天）
- 支持双周周期（例如第一周1357练、第二周246练）
- 每个周期日可单独设置：训练/休息、训练主题
- 当前周期周次 + 周期起始日 + 下次训练时间
- 模板化快速套用（如“两周1357/246”）

3. 训练记录面板

- 最近7天/30天训练历史
- 每次训练的动作、组数、次数、器械、重量、RPE
- 自动计算训练量与e1RM变化

4. 恢复与体征面板

- 睡眠时长、主观疲劳、疼痛评分、压力评分
- Deload风险提示

5. 饮食记录面板（先做轻量）

- 每日总热量
- 蛋白质/碳水/脂肪
- 饮水量
- 可选：体重、腰围

6. 体测记录库面板（原“知识库”重命名）

- 面板名称统一为：体测记录库（Body Composition Records）
- 该菜单聚焦体测单管理，不再承担通用知识库职责
- 支持上传体测单图片/PDF，按日期预览与检索
- 体测单源文件强制落在固定路径（不可编辑）
- 支持OCR提取并生成可编辑电子表单
- 基于电子表单生成身体变化趋势图表（体重/体脂率/肌肉量等）

## 2.5) 训练计划重设计（新增）

问题诊断：

- 旧版“按周一到周日”表单不直观，且难以表达双周或多周循环计划
- 用户无法直接编辑“周期中的第N天”训练状态
- 首页缺少“历史已练 + 未来待练”的连续时间视图

重设计方案：

1. 计划编辑模型从固定周计划升级为“周期日模板”

- 核心字段：`cycle_length_days`、`cycle_start_date`、`cycle_day_plan[]`
- 每个 `cycle_day_plan` 至少包含：day_index、is_training、focus_area

2. 训练计划页交互升级

- 可设置周期天数（7/14/21/28）
- 可批量重建模板，并逐日编辑训练/休息与训练主题
- 提供双周模板快捷入口（1357 / 246）

3. 首页新增训练日历视图

- 展示最近若干天 + 未来若干天（同屏）
- 状态区分：已完成、计划训练、未完成、休息
- 可快速回答“前几天练了没、后几天怎么练”

验收标准：

- 用户能在训练计划页完成双周周期录入并保存
- 首页日历能正确标注历史训练完成情况与未来训练安排
- 修改周期模板后，首页日历与下次训练展示同步更新

## 2.4) 首页目标达成反馈（新增）

目标配置字段：

- 目标类型：增肌 / 减脂 / 维持
- 起始日期、起始体重（kg）
- 目标日期、目标体重（kg）
- 起始肌肉量（kg，可来自体测单）
- 目标肌肉量（kg，可选）

首页反馈字段：

- 当前体重（kg）
- 距离目标体重差值（kg）
- 当前肌肉量（kg）
- 距离目标肌肉量差值（kg）
- 剩余天数
- 达成目标所需周变化量（kg/周）
- 实际周变化量（kg/周，基于最近28天趋势）
- 进度标签：健康 / 过慢 / 超额

判定规则（首版）：

- 差值计算：目标值 - 当前值（按目标方向展示“还差多少”）
- 所需周变化量 = 剩余差值 / (剩余天数 / 7)
- 实际周变化量 = 最近28天线性趋势斜率（按周换算）
- 进度标签判定：
  - 过慢：实际周变化量 < 所需周变化量的80%
  - 健康：实际周变化量在所需周变化量的80%到120%之间
  - 超额：实际周变化量 > 所需周变化量的120%

首页示例（增肌场景）：

- 当前65kg，目标7月到73kg
- 首页显示：目标还差8kg；若肌肉目标为+4kg，则显示肌肉还差4kg
- 系统同步显示：按剩余周数计算的所需周增重，并与实际周增重比较后输出健康/过慢/超额

## 2.3) 体测单固定路径与OCR流程（新增）

固定路径要求：

- 体测单源文件必须存放在固定路径，例如：`data/body_assessments/raw/`
- 该路径由系统配置，前端仅展示，不允许用户修改路径字符串
- 允许上传、替换、删除文件；不允许改动“存储目录规则”
- 体测记录库菜单下必须可见体测单文件列表（不可为空时给出上传引导）

OCR + 电子表单流程：

1. 用户上传体测单图片/PDF到固定路径
2. 系统触发OCR识别并提取结构化字段
3. 自动生成电子表单草稿（包含置信度标记）
4. 用户可逐字段纠正并保存（形成“校正后真值”）
5. 校正后数据进入趋势分析与可视化模块

电子表单核心字段（按体测单分区，首版）：

A. 基本信息区：

- 会员编号/姓名（如有）
- 身高（cm）
- 年龄
- 性别
- 测试日期与时间

B. 人体成分分析区：

- 身体总水分（L）
- 蛋白质（kg）
- 无机盐（kg）
- 体脂肪（kg）
- 体重（kg）

C. 肌肉脂肪分析区：

- 体重（kg）
- 骨骼肌（kg）
- 体脂肪（kg）

D. 肥胖分析区：

- BMI
- 体脂百分比（%）

E. InBody评分与控制区：

- InBody评分
- 目标体重（kg）
- 体重控制（kg）
- 肌肉控制（kg）
- 脂肪控制（kg）

F. 代谢与风险区：

- 腰臀比
- 内脏脂肪等级
- 去脂体重（kg）
- 基础代谢率（kcal）
- SMI（kg/m²）
- 建议热量摄入（kcal，体测单文案可保留“建议热零摄入”原词）

G. 补充区（可选）：

- 节段肌肉均衡（四肢/躯干）
- 节段脂肪分析（四肢/躯干）
- 生物电阻抗（各频段）
- 备注（OCR疑似错误、人工修正说明）

电子表单字段规则：

- 每个数值字段保存：`raw_text`（OCR原文）、`parsed_value`（解析值）、`unit`、`confidence`、`is_corrected`
- 用户修正后保留修正痕迹：`corrected_by`、`corrected_at`、`correction_reason`
- 时间序列图表使用 `parsed_value` 的最终修正值，不使用 `raw_text`

电子表单布局（仿体测单结构）：

1. 左侧主区域：人体成分分析、肌肉脂肪分析、肥胖分析、节段分析
2. 右侧摘要区域：InBody评分、体重控制、腰臀比、内脏脂肪、代谢指标
3. 底部趋势区域：历史记录折线图（体重/骨骼肌/体脂率）
4. 表单模式与原图模式可切换（并排查看，便于人工纠错）

图表展示（首页预览 + 体测记录库详情页）：

- 体重趋势线（按日期）
- 体脂率趋势线（按日期）
- 肌肉量趋势线（按日期）
- 去脂体重趋势线（按日期）
- 基础代谢率趋势线（按日期）
- 内脏脂肪等级趋势线（按日期）
- 关键阶段对比（最近一次 vs 4周前/8周前）
- 若数据缺失，图表仍展示并提示“待补齐字段”

## 2.2) 可编辑与AI代改机制（新增）

编辑原则：

- 用户可直接手动编辑所有非系统锁定字段
- AI只能“提议修改”，不能默认直接落库

AI代改流程：

1. AI识别到可改动项后生成“变更提议”
2. 前端展示变更前后对比（diff）
3. 用户明确同意（同意/拒绝）
4. 同意后写入数据库并记录审计日志

审计日志至少包含：

- 谁发起（user/ai）
- 改了什么字段
- 改前值、改后值
- 时间戳
- 同意凭据（按钮确认或命令确认）

高风险字段限制：

- 训练负荷上调超过阈值（如>5%）
- Deload相关决策
- 疼痛/伤病相关建议
  以上改动必须二次确认。

## 7.2) API控费策略（DeepSeek，30 RMB/月）

预算边界：

- 月预算上限：30 RMB
- 可用预算：27 RMB（预留 10% 应急）
- 软阈值：70%（达到后禁用高成本路径）
- 硬阈值：100%（达到后仅保留规则引擎）

调用分层：

1. L0 规则引擎（0 成本）

- 训练提醒
- 固定规则建议（加重/维持/减量）
- 模板化周报

2. L1 低成本模型（主力）

- 建议解释
- 日常问答

3. L2 高成本模型（严格限流）

- 复杂冲突决策
- 周期重排
- 高风险场景解释

Token 约束：

- 单次输入上限：1200 tokens
- 单次输出上限：300 tokens
- 上下文仅保留最近 3 次训练 + 最近 7 天恢复

RAG 约束：

- 仅复杂问题触发
- Top-K = 3
- 低预算阶段关闭重排器

降级策略：

- 达到软阈值：停用 L2，仅 L0 + L1
- 达到硬阈值：仅 L0（规则引擎 + 模板输出）

每周复盘指标：

- token 消耗
- 每用户成本
- 缓存命中率
- 建议采纳率
